# 원씽 "요청 주고받기"(1:1 메신저) 설계

작성일: 2026-06-25
대상 파일: `C:\AI\OneThing\index.html` (단일 파일 바닐라 JS 웹앱)

---

## 1. 개요 / 목적

원씽(할 일 노트) 앱에 **사용자 간 "요청 주고받기"** 기능을 추가한다.
로그인한 사람이 다른 등록 사용자에게 짧은 요청(메시지)을 보내면, 받는 사람의 원씽 페이지
**맨 아래 "📥 받은 요청" 섹션**에 그 요청이 나타난다. 받는 사람은 요청을 **내 할 일로 추가**하거나,
**완료/확인 표시**, **한 줄 답장**, **삭제**할 수 있고, 보낸 사람은 그 상태와 답장을 확인할 수 있다.

지금 당장은 1:1(한 명에게만 보내는) 상황이지만, 등록·선택 구조라 사람이 늘어도 그대로 동작한다.

## 2. 용어

- **로그인 / 사용자**: 요청을 주고받기 위한 "이름표". 이름 + 4자리 비밀번호로 등록·로그인.
- **동기화 코드**: 기존 기능. 같은 코드를 넣은 기기끼리 *한 사람의 노트(할 일)*를 공유. **이번 작업에서 건드리지 않는다.**
- **요청(request)**: 한 사람이 다른 사람에게 보내는 짧은 메시지 한 건.

## 3. 핵심 아키텍처 결정 — "분리"

로그인(사용자)과 노트 동기화는 **분리**한다.

- **노트(할 일) 동기화는 기존 그대로** — `notes/<동기화코드>` 경로, 변경 없음.
- **로그인은 요청 전용 이름표** — 노트가 로그인을 따라다니지 않는다.
- **요청 메시지는 노트와 별개의 Firebase 칸**(`users/`, `requests/`)에 저장 → 동기화 코드와 무관하게
  "로그인한 사람" 기준으로 배달된다.

### 중요한 구현 함의
현재 코드는 **동기화 코드가 설정됐을 때만** Firebase를 초기화한다(`initSync`).
요청 기능은 동기화 코드와 무관하므로, **로그인만 했다면 동기화 코드가 없어도 Firebase가 초기화**되어야 한다.
→ 페이지 로드시 `cloudConfigured()`면 Firebase 앱을 1회 초기화하고, 메시징 리스너는 로그인한 사용자 기준으로 붙인다.
노트 동기화 리스너(`notes/<코드>`)와 메시징 리스너(`requests/`)는 같은 `firebase.database()`를 공유하되 서로 독립적이다.

## 4. 데이터 모델 (Firebase Realtime DB 루트)

기존 `notes/` 와 나란히 두 개의 칸을 추가한다.

### 4.1 `users/<userId>`
```
userId: 생성된 고유 id (uid())            // 안정적 식별자
{
  name:      "나두혁",     // 표시 이름 (등록시 중복 불가)
  pin:       "1234",      // 4자리 비밀번호 (평문 — 5장 보안 항목 참고)
  createdAt: 1750000000000
}
```
- 등록된 사용자 목록 = `users` 전체. 로그인 화면의 "사람 고르기"와 요청 보내기의 "받는 사람 드롭다운"에 사용.

### 4.2 `requests/<reqId>`
```
reqId: uid()
{
  from:     "<senderUserId>",
  fromName: "나두혁",         // 보낸 사람 표시 이름 캐시
  to:       "<recipientUserId>",
  toName:   "양수민",
  text:     "월요일까지 보고서 부탁해요",
  ts:       1750000001000,   // 보낸 시각
  status:   "sent",          // "sent" | "read" | "done"
  replies:  [                 // 한 줄 답장(가벼운 양방향). 없으면 []
    { from: "<userId>", fromName: "양수민", text: "넵 확인했어요", ts: 1750000002000 }
  ],
  hiddenFor: []              // 이 목록에 든 userId에게는 화면에서 숨김(=그 사람이 "삭제")
}
```

- **요청 1건 = 칸 1개**를 보낸 사람·받는 사람이 *함께* 본다 → 상태/답장이 양쪽에 실시간 반영.
- 클라이언트는 `requests` 전체를 구독하고, 화면에서 **나와 관련된 것만** 필터한다:
  - 받은 요청: `to == 내 userId` 그리고 `hiddenFor`에 내 id 없음
  - 보낸 요청: `from == 내 userId` 그리고 `hiddenFor`에 내 id 없음
  - (2명 규모에선 전체 구독이 가장 단순·안정적. 사람이 많이 늘면 추후 인덱싱 고려 — 이번 범위 밖.)

### 4.3 로컬(이 기기) 저장
- `ONETHING_USER` (localStorage): 현재 로그인한 `userId`. 있으면 자동 로그인(기억).

## 5. 상태 흐름 (status)

```
sent (보냄)  ──받는 사람이 화면에서 봄──▶  read (읽음)  ──받는 사람이 [완료/확인]──▶  done (완료)
```

- **안 읽음** = `to == 나 && status == "sent"`. 상단 배지에 개수 표시(`📥 2`).
- 받는 사람의 화면에 받은 요청 카드가 보이는 순간(또는 섹션 펼침 시) `sent → read`로 올린다.
- `[완료/확인]` 버튼 → `read/sent → done`. 보낸 사람 화면의 "보낸 요청"에 "완료"로 표시.
- 답장은 status와 별개로 `replies`에 추가(양쪽 다 추가 가능, 가벼운 스레드).

## 6. 화면 & UX 흐름

### 6.1 로그인 게이트 (첫 진입 / 사용자 바꾸기)
- 페이지 로드시 `ONETHING_USER`가 없거나 그 사용자가 `users`에 없으면 **로그인/등록 화면**(오버레이)을 띄운다.
- 구성:
  - **사람 고르기**: 등록된 사용자 목록 → 고르면 4자리 비번 입력 → 일치시 로그인.
  - **+ 새로 등록**: 이름 + 4자리 비번 입력 → 등록 후 자동 로그인.
- 로그인하면 `ONETHING_USER` 저장(이 기기 기억). 다음 방문부터 바로 할 일 화면.

### 6.2 상단 표시
- 사용자 칩: `👤 나두혁 ▾` → 클릭 시 **사용자 바꾸기 / 로그아웃**.
- 안 읽음 배지: `📥 2` (없으면 숨김). 클릭 시 받은 요청 섹션으로 이동/포커스.
- `✉️ 요청 보내기` 버튼.

### 6.3 요청 보내기
- `✉️ 요청 보내기` → 작은 폼/모달:
  - **받는 사람**: 등록된 사용자 드롭다운(나 자신 제외).
  - **내용**: 한 줄~여러 줄 텍스트.
  - 보내기 → `requests/<reqId>` 생성(status `sent`).

### 6.4 📥 받은 요청 (페이지 맨 아래 — 첨부 이미지 위치)
- 카드마다: 보낸사람 · 시각 · 내용 + 버튼:
  - `[내 할 일로 추가]` → 요청 내용을 `state.todos`에 추가(기존 노트 동기화로 다른 기기에도 반영). 요청 자체는 남음.
  - `[완료/확인]` → status `done`.
  - `[↩ 답장]` → 한 줄 입력 → `replies`에 추가.
  - `[삭제]` → 내 `hiddenFor`에 내 id 추가(내 화면에서만 사라짐).
- 안 읽은 카드는 강조(점/굵게 등).

### 6.5 📤 보낸 요청 (작은 목록)
- 내가 보낸 요청의 status(보냄/읽음/완료)와 상대 답장 확인.
- `[삭제]` → 내 `hiddenFor`에 추가(내 화면에서만).

## 7. i18n

- 새로 생기는 모든 문구는 기존 `I18N` 사전 + `data-i18n*` 방식으로 한/영 둘 다 제공(기존 🌐 전환과 일관).
- 사용자가 입력한 이름·요청 내용·답장은 번역 대상 아님(원문 그대로).

## 8. 예외 처리

- **이름 중복 등록**: 같은 이름이 `users`에 있으면 새로 못 만들게 막고 "본인이면 목록에서 골라 비번으로 로그인"하도록 안내.
- **비번 불일치**: 로그인 거부 + 안내 메시지.
- **오프라인 / Firebase 미연결**: 노트는 기존처럼 로컬 저장. 요청 전송 시 연결 안 됐으면 토스트로 안내(연결되면 반영). 메시징은 Firebase 설정이 있어야 동작 — 설정 자체는 파일에 박혀 있으므로 인터넷만 있으면 동작.
- **받는 사람이 0명**(나 외 등록자 없음): "보낼 상대가 없습니다 — 상대가 먼저 등록해야 해요" 안내.
- **상대가 요청을 삭제(hiddenFor)**: 노드는 남지만 그 사람 화면에서만 숨김 → 보낸 사람에게 갑자기 사라지는 혼란 없음.

## 9. 보안 현실 / 한계 (명시)

- 4자리 비번 + 공개 Realtime DB(읽기/쓰기 열림)라, 이는 **친한 동료끼리 '구분'하는 가벼운 잠금**이지
  강력한 보안이 아니다. 비번·요청 내용이 클라우드에 평문으로 저장된다.
- → **민감한 정보(개인정보·비밀)는 넣지 않는다**는 전제. 사용자도 이 한계를 이해함(이름+4자리로 충분하다고 요청).
- 강화가 필요해지면(추후, 범위 밖): Firebase 보안 규칙 + 인증, 비번 해시 등.

## 10. 범위 밖 (YAGNI)

- 단체방/그룹 발송, 파일·이미지 첨부, 푸시/알림음, 요청 검색·정렬, 읽음 시각 상세 로그.
- 비번 찾기/변경 UI(필요해지면 별도). v1은 등록·로그인·요청·답장·상태·삭제(숨김)까지.

## 11. 성공 기준 (검증 가능)

1. 새 기기에서 처음 열면 로그인/등록 화면이 뜬다.
2. 이름+4자리로 등록되고 `users/<id>`에 저장된다. 같은 이름 재등록은 막힌다.
3. 로그인 후 상단에 이름이 표시되고, 새로고침해도 로그인이 유지된다(이 기기 기억).
4. A로 로그인 → B에게 요청 전송 → B로 로그인한 기기의 "받은 요청"에 표시된다.
5. 받은 요청 `[내 할 일로 추가]` 시 왼쪽 할 일 목록에 추가된다.
6. `[완료/확인]` 시 보낸 사람(A) 화면의 "보낸 요청"에 '완료'로 보인다.
7. `[↩ 답장]` 시 A의 "보낸 요청"에 답장이 보인다.
8. 안 읽은 요청 개수 배지가 정확히 표시된다.
9. 🌐 전환 시 새 UI 문구도 한/영 번역된다.
10. 기존 동기화 코드/노트 데이터·동작에 영향이 없다(회귀 없음).

## 12. 가정 / 미해결

- Firebase 설정값은 파일에 이미 박혀 있고 DB 규칙이 열려 있다고 가정(기존 동기화가 그렇게 동작 중).
- "한 줄 답장"은 `replies` 배열로 가벼운 스레드까지 허용(과한 대화 UI는 만들지 않음).
- 받는 사람 선택은 등록된 사용자 드롭다운(이름 직접 타이핑 없음).
